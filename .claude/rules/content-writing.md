# コンテンツ執筆品質ルール

このルールは、ブログ記事の執筆品質に関するガイドラインを定義する。

## 適用条件

- `content/posts/**/*.md` ファイルの作成・編集時

---

## 1. ブロック要素のフリ（導入文）

引用ブロック・図・リンクカード・表・コードブロックの前には**フリ（導入文）を必ず置く**。フリがないと読者は「なぜこれが今ここにあるのか」を把握できない。

```markdown
<!-- Good: フリがある -->
公式ドキュメントでは次のように説明されています。

> Pagefind is a fully static search library...

<!-- Bad: フリがない -->
## Pagefindとは

> Pagefind is a fully static search library...
```

**省略してよい場合**: 直前の文がそのまま導入として機能している場合（例: 「`astro.config.mjs` に以下を追記します。」→ コードブロック）

## 2. 引用の設計

- 引用の前に「誰が何について述べているか」を一文で書く
- 自分の主張を先に述べ、引用はその根拠として置く
- 前後の文で既に伝わっている内容を繰り返すだけの引用は削除する

## 3. リンクカードの導入

- リンクカード（`<URL>` 単体行）の前に「何のリンクか」を一文で書く
- 参考セクションに並べるリンクカードは、セクション見出しがフリの役割を果たすため個別のフリは不要


## 4. 事実と評価の分離

選定理由やメリットの列挙では、検証可能な事実と主観的評価を同列に並べない。

```markdown
<!-- Bad: 事実と曖昧な評価が混在 -->
- 外部サービス不要
- クライアント側の追加読み込みが不要
- 静的サイトジェネレーターとの相性が非常に良い  ← 「非常に」の基準が不明

<!-- Good: 事実のみ列挙し、主観は主観表現を付ける -->
- 外部サービス不要
- クライアント側の追加読み込みが不要
- 静的サイトジェネレーターとの相性が良いと感じました  ← 主観を明示
```

- 主観的評価には「〜と感じました」「私の印象では」等の表現を付けて書く
- 事実の列挙に混ぜず、感想セクションに分離することも検討する
