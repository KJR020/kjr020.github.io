今回はHugoで構築したブログをAstroに移管した話です。

## なぜAstroにしたか
ポートフォリオも兼ねて作成したかったので、ある程度カスタマイズがしやすいものを探していた。
Hugoでもカスタマイズは可能だったが、使い慣れていないGo言語でテンプレートを書く必要があり、
Reactを使用して構築したかった。

ChatGPTに相談しつつ前提となる条件を整理した。

- 前提・仮定（記事に残す用）
	- ホスティングは GitHub Pages（GitHub.io）（= 静的ファイル配信のみ）
	- 目的は ブログ運用 + ポートフォリオ（凝った実装で技術力を見せる） の両立
	- 「凝った実装」= Reactコンポーネント駆動 / インタラクション / デモページ（ミニアプリ）を作りやすいこと
	- 記事運用は Markdown中心で軽く（= 記事追加が面倒だと継続できない）
	- SEOは（少なくとも最優先ではない）※必要なら後で追加可能
	- できれば ビルド/デプロイをGitHub Actionsで自動化し、pushで更新できる状態にしたい

上記の観点を踏まえて、DeepResearchを活用しつつ、ブログ構築に使えそうなフレームワークを調査した。
その結果、以下のような選択肢が挙がった。

- **Docusaurus（React + MDX）**
  - 採用理由
    - ブログ機能が最初から揃っていて、運用が軽い
    - MDXで「記事内にReactコンポーネント」を埋め込める → デモを載せやすい
    - GitHub Pages前提のデプロイ導線が明確
  - 捨て理由（ありうる）
    - 情報設計が「ドキュメント/ブログ」寄りで、完全自由なサイト設計はしにくい

- **Astro + React（“Reactベース”ではないがReactを主戦力にできる）**
  - 採用理由
    - Markdown運用を軽く保ちつつ、必要なところだけReactで凝れる
    - ポートフォリオ（作品集/デモ）とブログを同居させやすい
    - 静的出力が基本でGitHub Pagesと相性が良い
  - 捨て理由（ありうる）
    - 「Reactベース」に厳密にこだわると定義上は外れる

- **Gatsby（React + GraphQL）**
  - 採用理由
    - Reactで自由に作り込める、ブログテンプレも多い
  - 捨て理由（今回の文脈で強い）
    - GraphQL（ビルド時）が絡み、記事運用や構造変更の認知負荷が増えやすい  
      → 「凝ったUIを作りたい」のに別の難所が増える

- **Next.js（静的エクスポート）**
  - 捨て理由
    - Markdown運用の導線は自前構成になりがち（手間が増えやすい）
    - GitHub Pagesでの静的運用は制約・ハマりどころが多い
- **Hugo（現状）**
  - 捨て理由（今回の主因）
    - Goテンプレ中心で、コンポーネント駆動UI/インタラクションの開発体験が弱い
