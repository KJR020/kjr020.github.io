## High指摘の詳細

### K-3: descriptions配列によるプライベート情報漏洩リスク

**リスクレベル**: High

**発見事項**: 設計書の最重要リスク。新規エンドポイントは **descriptions配列をraw形式（最大5要素）** でクライアントに返す設計。Scrapboxの `descriptions` はページの先頭5行であり、体系タグ以外に以下のような情報が含まれうる:
- 個人的なメモ・感想
- 社内プロジェクト名や顧客名
- URLやアクセス情報
- 未公開の技術的アイデア

要件書では「Scrapboxページの本文内容のブログ上での表示」を非目標としているが、descriptions配列の生データ公開はこの原則と矛盾する。

**推奨対策**: 以下のいずれかを採用:
1. **サーバーサイドでタグ抽出**（推奨）: Proxy側で `descriptions` から体系タグ・深さタグのみを抽出し、クライアントには `tags: ParsedTaxonomyTag[]` と `depthLevel` のみ返す。descriptions原文は返さない
2. **クライアント返却前にフィルタリング**: descriptions配列のうち `[体系/...]` `[深さ/...]` パターンに一致する行のみ返す
3. **リスク許容**: 個人ブログかつAPIアクセスはサイト訪問者に限定されることを前提にリスクを受容する。ただし設計書にリスク受容の判断と根拠を明記すること

### K-8: 18リクエスト並列のScrapbox API呼び出しに対するレート制限

**リスクレベル**: High

**発見事項**: 1,788ページを100件ずつ18バッチで `useQueries` により並列フェッチする設計。これにより:
1. **Scrapbox API側のレート制限**: 1ユーザーの1回のページロードで18リクエストが同時にScrapbox APIに送信される。Scrapbox APIのレート制限は公式に文書化されていないが、大量リクエストはブロックされる可能性がある
2. **Proxy側のリソース消費**: Cloudflare Pages Functionsの同時実行数制限
3. **悪意あるリフレッシュ**: ユーザーが `refetch()` を連打した場合の増幅

**推奨対策**:
- **サーバーサイド**: Proxy側でページネーションを集約するエンドポイント設計を検討（1リクエストで全ページ返す）
- **クライアントサイド**: 並列度を3-5に制限
- **デバウンス**: `refetch()` にクールダウン（例: 30秒間隔制限）を設定
- **Proxy側レート制限**: `_middleware.ts` でIPベースのレート制限

## Medium指摘の詳細

### K-1: 新規Proxyエンドポイントでの認証情報の取り扱い

設計書では新規エンドポイント追加を明記しているが、認証ロジックの記載がない。既存の `[project].ts` と同じ `SCRAPBOX_SID` → `Cookie` ヘッダー注入パターンを踏襲すべき。エラーレスポンスに内部情報を含めないことを明記する。

### K-5: YAMLパース時のセキュリティ

`yaml` パッケージはデフォルトで安全なスキーマを使用するが、YAML爆弾への耐性は利用側で担保が必要。
```typescript
import { parse } from "yaml";
const data = parse(yamlContent, { maxAliasCount: 100 });
const framework = FrameworkDefinitionSchema.strict().parse(data);
```

### K-6: クエリパラメータのバリデーション不足

`limit` を `1-100` の整数、`skip` を `0` 以上の整数に制限するバリデーション。クエリ文字列の透過は避け、バリデーション済みパラメータのみでUpstream URLを構築する。

### K-9: CORSポリシー未定義

レスポンスに `Access-Control-Allow-Origin: https://kjr020.github.io` を設定。`_middleware.ts` でOriginチェックを一元管理。既存エンドポイントも含めて対応すべき。

### K-10: Cache-Controlヘッダー未設定

`Cache-Control: public, max-age=300, s-maxage=300` を設定。18リクエストの各レスポンスにキャッシュが効けば2回目以降の負荷を大幅に削減。

### K-11: XSSリスク

ReactのJSX内テキストレンダリングは自動エスケープされるため通常は安全。`dangerouslySetInnerHTML` を使用しないことを設計方針として明記。テストでXSSペイロード文字列を含むタイトルの表示テストを追加。

## 総合評価

設計書のセキュリティ方針は既存のProxy実装と整合性があり、基本的な防御（認証情報のサーバーサイド管理、パストラバーサル対策、GETメソッド限定）は踏襲されている。

**2つのHigh指摘**が実装前に解決を要する:
1. **K-3**: descriptions全行を公開する必要はない。サーバーサイドでタグを抽出してから返す設計に変更すべき
2. **K-8**: 18並列リクエストは既存Proxyの設計前提を大きく超えており、並列度制限またはProxy集約エンドポイントの設計が必要
