# 機能仕様書: Scrapbox知識体系管理システム

> **ステータス**: ドラフト
> **作成日**: 2026-03-25
> **著者**: KJR + Claude

## 概要

Scrapboxに日常的にメモを書くだけで、自分の知識が「点→線→面」へと育っていく仕組みを作る。単発の知識の蓄積ではなく、知識同士の接続と応用を促す**知識体系**を構築し、ブログ上で俯瞰ビューとして可視化する。

## 背景と課題

KJRはScrapboxにソフトウェアエンジニアリング、料理、語学、ビジネスなど多岐にわたるメモを書いている。ハッシュタグやリンクで緩く分類してはいるものの、以下の課題がある:

- **知識が点のまま**: 個別のページは増えるが、それらがどう繋がるのか見えない。学んだことを応用する「スキーマ」が形成されにくい
- **全体像が見えない**: 自分がどの領域に強くて、どこが足りないのか俯瞰できない。次に何を学ぶべきかの判断材料がない
- **体系がない**: 知識の整理は自分の感覚頼み。その分野の「強い人」がどんな知識体系を持っているのか、比較する基準がない

## このシステムが目指す世界

KJRが毎日Scrapboxにメモを書く。それだけで以下が自然に起きる:

1. 書いたメモが知識体系の中に位置づけられる
2. 既存の知識との接続が見える化される（点→線）
3. ある領域の知識がまとまると「面」として応用可能になったことが分かる
4. 参照フレームワークと比較して、足りない部分が一目で分かる

## シナリオ

### シナリオ1: 知識体系の初期構築 — 「強い体系をパクる」

KJRは「ソフトウェアエンジニアリング」ドメインの知識を体系化することにした。

まず、「参照フレームワーク」を選ぶ。KJRはroadmap.shのバックエンドロードマップが自分の学びたい領域と近いと感じ、これをベース体系として取り込む。取り込みは、プロジェクトのYAML設定ファイルに参照フレームワークの構造（カテゴリ階層）を定義することで行う。

```yaml
# 例: frameworks/software-engineering.yaml
name: ソフトウェアエンジニアリング
source: roadmap.sh Backend Developer Roadmap
categories:
  - name: インターネット
    children: [HTTP, DNS, ブラウザの仕組み]
  - name: プログラミング言語
    children: [TypeScript, Python, Go, Rust]
  - name: データベース
    children: [PostgreSQL, Redis, MongoDB]
  # ...
```

次に、ブログの知識マップページを開く。ロードマップの各カテゴリがノードとして表示されるが、まだScrapboxのページとの紐付けがないので、ほとんどがグレーアウトしている。

KJRはScrapboxの既存ページを確認する。「TypeScript」「PostgreSQL」「Docker」などのページがある。これらのページの中に `[体系/SE/言語/TypeScript]` というScrapboxリンクを追加する。これがタクソノミーへの紐付けになる。

知識マップを更新すると、紐づいたノードが色づく。「プログラミング言語」カテゴリのカバレッジが上がり、「セキュリティ」や「テスト戦略」のカテゴリは空のままだと分かる。KJRは「次はテスト戦略について集中的に学ぼう」と学習計画を立てる。

### シナリオ2: 日常のメモ書きで体系が育つ — 「継続的な成長」

ある日、KJRはTerraformのモジュール設計パターンについて学んだ。Scrapboxに「Terraformモジュール設計パターン」というページを作り、いつも通りメモを書く。

メモの中で、KJRは自然に `[Terraform]` `[IaC]` `[ディレクトリ構造]` といったScrapboxリンクを書く。さらに、以前書いた `[クリーンアーキテクチャ]` との類似性に気づき、そのリンクも追加する。ページ末尾に `[体系/インフラ/IaC/Terraform]` を書いて体系に紐づける。

ブログの知識マップを見ると、2つのことが見える:

1. **体系内の位置**: 「インフラ > IaC > Terraform」のノードにページが追加された。IaCカテゴリはTerraformだけなので、Pulumiなど他のツールが足りないことも分かる
2. **知識の接続**: 「Terraformモジュール設計パターン」は「クリーンアーキテクチャ」と接続されている。つまりインフラの知識がソフトウェア設計の知識と「線」で繋がった

こうした接続の蓄積が「面」を形成し、KJRは「設計原則はインフラにも適用できる」というスキーマを獲得していく。

### シナリオ3: 複数の体系を組み合わせる — 「アレンジ」

SEドメインがある程度育ったので、KJRは「料理」ドメインにも着手する。

料理の参照フレームワークとして、基本的な料理分類（和食・洋食・中華、調理法: 焼く・煮る・蒸す・揚げる）と、「料理の四面体」（火・水・油・空気の理論）を追加する。

Scrapboxに「鶏むね肉の低温調理」というページを書き、`[体系/料理/調理法/煮る]` `[体系/料理/素材/鶏肉]` と紐づける。さらに、このページで温度管理のプロセスについて書いているうちに、`[プロセス最適化]` という概念がSEの `[CI/CDパイプライン]` と共通していることに気づく。

知識マップ上で、「料理 > 調理法」と「SE > DevOps」がドメインをまたいで接続される。KJRは「プロセスの最適化という考え方は、料理にもソフトウェアにも共通する汎用スキルだな」と理解する。これが「複数体系の組み合わせ」の力。

### シナリオ4: ポートフォリオとしての閲覧

鈴木さんがKJRのブログを訪れ、ナビゲーションから「Knowledge Map」ページを開く。

画面にはドメインごとのカバレッジサマリーが表示される:
- ソフトウェアエンジニアリング: 32/48ノード (67%)
- インフラ: 12/20ノード (60%)
- 料理: 8/15ノード (53%)
- 語学: 5/12ノード (42%)

「ソフトウェアエンジニアリング」をクリックすると、サブカテゴリがツリーマップやグラフとして展開される。各ノードには紐づいたScrapboxページの数と最終更新日が表示される。鈴木さんは「この人はバックエンドとインフラに強くて、最近はAI/ML方面も広げているんだな」と直感的に把握する。

### シナリオ5: ギャップ分析と学習計画

月に1回、KJRは知識マップの「ギャップ分析」ビューを開く。

このビューでは、参照フレームワークのカテゴリのうち、カバレッジが低い（Scrapboxページが少ない or 古い）領域がハイライトされる。さらに、「接続度」も表示される — そのカテゴリの知識が他のカテゴリとどれだけ繋がっているか。

KJRは以下を見る:
- 「セキュリティ」: カバレッジ20%、接続度低 → 孤立した知識。他の領域と繋がっていない
- 「テスト戦略」: カバレッジ40%、接続度高 → 知識はあるが網羅性が足りない
- 「料理 > 発酵」: カバレッジ0% → まだ手つかずの領域

KJRはこの分析結果をもとに、「今月はセキュリティの知識を増やし、特にDevOpsや設計パターンとの接続を意識して学ぼう」と計画を立てる。

### 構成要素

**1. 参照フレームワーク定義**
- YAMLファイルでドメインごとの知識体系を定義
- 「強い体系」のベース（roadmap.sh、SWEBOK、料理分類等）をここに記述
- 複数のフレームワークを追加・組み合わせ可能

**2. Scrapboxページ解析**
- 既存のAPI Proxyを拡張し、ページ詳細（リンク情報含む）を取得
- `[体系/ドメイン/カテゴリ/...]` 形式のリンクをパースしてタクソノミーに紐付け
- ページ間のリンクを抽出して接続（線）を構築

**3. 知識グラフ構築**
- 参照フレームワーク × Scrapboxページのマッピング
- ページ間の接続情報からグラフを構築
- カバレッジ（網羅度）と接続度（応用可能性）を算出

**4. ブログ上の知識マップページ**
- ドメイン選択 → カテゴリツリー/グラフ表示
- カバレッジのヒートマップ（色の濃淡でカバー度合いを表現）
- 知識グラフの可視化（ノード=ページ、エッジ=リンク）
- ギャップ分析ビュー（足りない領域のハイライト）

## Scrapbox側の運用ルール

知識体系を育てるために、Scrapboxでの以下の運用ルールを導入する。既存のメモ習慣を大きく変えないことが最優先。

### 体系タグの記法

```
[体系/ドメイン名/カテゴリ/サブカテゴリ]
```

例:
- `[体系/SE/言語/TypeScript]`
- `[体系/SE/設計/クリーンアーキテクチャ]`
- `[体系/料理/調理法/低温調理]`
- `[体系/語学/英語/文法]`

**ルール**:
- ページに1つ以上の体系タグを付ける（複数可）
- 体系タグはページ本文の任意の場所に書いてよい（末尾推奨）
- 体系タグを付けなくてもページは存在できる（無理に分類しない）

### ページ間リンク（自然な接続）

Scrapboxの通常のリンク記法 `[ページ名]` がそのまま知識の接続になる。これは既存の使い方そのまま。

```
Terraformのモジュール設計は[クリーンアーキテクチャ]の考え方と似ている。
[ディレクトリ構造]の設計原則が適用できる。
```

### 知識の深さレベル（オプション）

Bloomのタクソノミーを参考に、ページに知識の深さタグを付けることもできる:

- `[深さ/知っている]` — 概念を説明できる
- `[深さ/使える]` — 実際に使った経験がある
- `[深さ/応用できる]` — 他の文脈にも適用できる
- `[深さ/教えられる]` — 他者に説明・指導できる

これはオプション。付けなくても体系は機能する。

## 非目標（やらないこと）

1. **Scrapboxの代替を作ること** — 知識の蓄積・編集はScrapboxで完結する。ブログ側に編集機能は持たない
2. **完全自動分類** — AIによるページの自動分類は行わない。体系への紐付けはユーザーが意図的に行う（ただし紐付け候補の提案は将来検討）
3. **リアルタイム同期** — Scrapboxの更新がブログに即座に反映される必要はない。定期的な更新（手動リフレッシュ or 定期ビルド）で十分
4. **他人の知識体系の管理** — マルチユーザー機能は不要。KJR個人の知識体系のみ
5. **知識の内容表示** — ブログ上ではページのタイトル・メタデータ・接続のみ表示。Scrapboxページの本文は表示しない（プライベート情報を含むため）
6. **全ページへの体系タグ強制** — タグなしのページも許容する。分類は段階的に進める

## 未解決事項

### 1. 参照フレームワークの初期セット
各ドメインの「強い体系」として何を採用するかは未決定。候補:
- SE: roadmap.sh, SWEBOK, またはその組み合わせ
- 料理: 基本的な料理分類、料理の四面体
- 語学: CEFR（欧州共通参照枠組）
- ビジネス: MBA基礎、ドラッカーの体系

**決定時期**: 機能仕様書の承認後、要件定義フェーズで確定

### 2. 可視化ライブラリの選定
知識グラフの可視化に使うライブラリ（D3.js, vis.js, Cytoscape.js, React Flow等）は設計フェーズで決定。

**決定時期**: spec-designフェーズ

### 3. API Proxyの拡張範囲
ページ詳細API（リンク情報含む）の取得を追加する必要がある。全ページの詳細を取得するとAPIコール数が多くなる。バッチ取得の方法やキャッシュ戦略は検討が必要。

**決定時期**: spec-designフェーズ

### 4. 体系タグの記法の最終形
`[体系/SE/言語/TypeScript]` 形式は案。Scrapboxのリンク文化により自然な記法がないか検討。例えばハッシュタグ `#SE/言語/TypeScript` を使う案もある。

**決定時期**: 要件定義フェーズで、既存のScrapboxページの記法を調査して決定

### 5. ビルドタイム vs ランタイム
知識グラフのデータをビルド時に生成するか、ランタイムにAPIから取得するかは設計判断。ビルド時のほうがパフォーマンスは良いが、鮮度は下がる。

**決定時期**: spec-designフェーズ

## 補足メモ（開発者向け）

### Scrapbox API で取得可能な情報

| エンドポイント | 取得データ | 知識体系への用途 |
|---------------|-----------|-----------------|
| `GET /api/pages/:project` | ページ一覧（タイトル、説明、更新日、linked数） | カバレッジ算出の基礎データ |
| `GET /api/pages/:project/:title` | ページ詳細（本文、リンク先一覧、被リンク） | 知識グラフの接続情報 |
| `GET /api/pages/:project/search/titles` | タイトル検索 | 体系タグのあるページの抽出 |

### 既存システムとの関係

- **Scrapbox API Proxy** (`.kiro/specs/20260322-scrapbox-api-proxy/`): 現行のProxy機能を拡張して、ページ詳細取得に対応する必要がある
- **ScrapboxCardList/Carousel**: 既存のカード表示はそのまま残す。知識マップは別ページとして追加
- **useScrapboxData hook**: 既存hookを拡張するか、知識体系用の新規hookを追加

### 段階的リリースの推奨

1. **Phase 1**: 参照フレームワーク定義 + 体系タグパーサー + カバレッジ表示（ツリーマップ）
2. **Phase 2**: ページ詳細API取得 + 知識グラフ可視化（接続線）
3. **Phase 3**: ギャップ分析ビュー + 接続度スコア
4. **Phase 4**: 複数フレームワークの組み合わせ + ドメイン横断接続
