## Introduction

Scrapboxに蓄積された1,788ページの知識メモを、ブログ上の `/knowledge` ページで**俯瞰・可視化**するシステムを構築する。既存のハッシュタグとリンク構造をそのまま活用し、新しいタグ記法やルールは導入しない。

### 設計方針の変更（変更履歴）

#### 変更1: 参照フレームワークYAML定義 → 削除

当初設計ではYAMLファイルで参照フレームワーク（SWEBOK等）を定義し、カバレッジやギャップ分析をブログ上で自動化する構想だった。レビューの結果、以下の方針に変更した:

- **参照フレームワークYAML定義 → 削除**: ギャップ分析はClaude Codeスキルで対話的に行う
- **体系タグ（`[体系/SE/...]`）→ 導入しない**: 既存のハッシュタグ・リンクをそのまま活用する
- **カバレッジ算出エンジン → 削除**: 比較基準となるフレームワークが不要になったため
- **ギャップ分析ビュー → 削除**: Claude Codeスキルに移管

**変更理由**: YAML定義の管理コストが「継続的に育てられる仕組み」という成功基準に反する。

#### 変更2: ページリンクグラフ (Req 5/6/7) → タグ共起グラフ (Req 8) に変更

当初はPhase 2でページ間リンク情報を使ったグラフ可視化を計画していたが、実装・試用後に撤回した。

**撤回した案 (Req 5/6/7) の問題点:**
- 1,788ページ分の詳細APIを叩くコストが高い (並列度3で最大60秒超、Cloudflare Functionsのタイムアウトに抵触しうる)
- 1,788ノードを描画するとハリネズミ状になり「どこが厚い/薄い」の判断ができない
- ページ単位の粒度では「この分野の知識が少ない」という俯瞰が得られない

**採用: タグ共起グラフ (Req 8)**
- 概念（タグ）レベルに集約することで、ノード数が視認可能な規模に収まる
- タグのページ数をノードサイズに反映することで、知識量の濃淡が一目で見える
- ページ詳細API呼び出しが不要なため、既存のKnowledge Proxyだけで成立する

**Req 8 の由来**: 当初Req 7の代替として導入。`tasks.md` のPhase 2タスクは撤回、Phase 1の機能として統合。

### 4つの原則との対応

| 原則 | 実現方法 |
|------|---------|
| 1. 体系として構築 | 既存タグ・リンクを可視化し、構造を「見える化」する |
| 2. 強い体系をベースに | Claude Codeスキルで参照フレームワークと対話的に比較 |
| 3. 俯瞰しギャップを補完 | ブログで俯瞰、ギャップ分析はClaude Codeスキルで |
| 4. 複数体系を組み合わせる | 知識グラフでドメイン横断の接続を発見 |

### 段階的リリース戦略

| Phase | スコープ | 含む要件 |
|-------|---------|---------|
| Phase 1 | 全ページ取得 + タグ抽出 + ドメイン別一覧 + タグ共起グラフ | Req 1, 2, 3, 4, 8 |
| ~~Phase 2~~ | ~~ページ詳細API + ページリンクグラフ~~ | **撤回** (Req 5, 6, 7) |

### 用語定義

| 用語 | 定義 |
|------|------|
| **ハッシュタグ** | Scrapboxページ内の `#タグ名` 形式のテキスト。ドメイン分類の基準として使用 |
| **リンク** | Scrapboxページ内の `[ページ名]` 形式の参照。知識グラフのエッジになる |
| **ドメイン** | ハッシュタグに基づくページの分類グループ（例: #書籍, #料理, #セキュリティ） |
| **知識グラフ** | ページ（ノード）とページ間リンク（エッジ）で構成されるグラフ構造 |

## Requirements

### Requirement 1: 既存タグ・リンク抽出

**Phase**: 1 — データモデル

**Objective:** As a KJR, I want Scrapboxページの既存ハッシュタグが自動的に抽出されること, so that 新しいルールを導入せずに知識を分類できる

#### Acceptance Criteria

1. システムはScrapboxページのdescriptions（先頭5行）からハッシュタグ（`#タグ名` 形式）を抽出すること
2. 抽出はサーバーサイド（Proxy）で行い、descriptions原文はクライアントに返さないこと [K-3]
3. 1つのページに複数のハッシュタグがある場合、すべて抽出すること
4. ハッシュタグのないページも一覧に含めること（「未分類」として扱う）
5. ハッシュタグの一覧（ユニークなタグとページ数）をフロントエンドに提供すること
6. `[...]` 内（URL fragmentやページリンク）に含まれる `#` はタグとして扱わないこと
7. `#` の直前は空白または行頭でなければならず、単語途中の `#` (例: `word#notTag`) はタグとして扱わないこと
8. タグ名の `_` はスペースに正規化すること（Scrapbox慣例: `#Claude_Code` → "Claude Code"）

### Requirement 2: ドメイン別一覧表示

**Phase**: 1 — フロントエンド

**Objective:** As a ブログ読者, I want KJRの知識をドメイン別に俯瞰できること, so that どんな分野に知識があるか直感的に把握できる

#### Acceptance Criteria

1. 知識マップページにハッシュタグごとのページ数サマリーが一覧表示されること
2. タグをクリックすると、そのタグが付いたページのリストが展開されること
3. 各ページにはタイトル、最終更新日、被リンク数が表示されること
4. タグの並び順はページ数の多い順（降順）であること
5. ページはレスポンシブデザインに対応し、モバイルでも閲覧可能であること
6. 「未分類」カテゴリが存在し、ハッシュタグのないページを含むこと

### Requirement 3: 知識マップページ

**Phase**: 1 — フロントエンド

**Objective:** As a ブログ読者, I want 知識の全体像を1ページで把握できること, so that KJRの知識の幅と深さを直感的に理解できる

#### Acceptance Criteria

1. ブログに `/knowledge` パスで知識マップページが存在すること
2. ページにはドメイン別一覧（Req 2）と、Phase 2で追加される知識グラフの両方を配置できるレイアウトであること
3. ページ全体の統計情報（総ページ数、タグ数、最終更新日）が表示されること
4. ナビゲーション（Header）に `/knowledge` リンクが追加されること

### Requirement 4: Knowledge Proxy（全ページ取得）

**Phase**: 1 — API

**Objective:** As a システム, I want Scrapbox全ページのタグ・メタデータを安全に取得できること, so that フロントエンドでドメイン分類と一覧表示に利用できる

#### Acceptance Criteria

1. `/api/knowledge/{project}` エンドポイントがクライアントからの単一リクエストで全ページデータを返すこと
2. Proxy内部でScrapbox APIをページネーション（100件ずつ）して全ページ取得すること
3. Scrapbox APIへの並列度は3以下に制限すること [K-8]
4. レスポンスにはdescriptions原文を含めず、抽出済みハッシュタグのみ含めること [K-3]
5. 既存のセキュリティ要件（認証情報の非露出、プロジェクト名バリデーション、GETメソッド限定）を継承すること
6. `Cache-Control: public, max-age=300` ヘッダーを付与すること
7. `Access-Control-Allow-Origin` を本番ドメインに制限すること [K-9]

### Requirement 8: タグ共起グラフ

**Phase**: 1 — フロントエンド

**Objective:** As a KJR, I want タグ同士の共起関係と知識量の濃淡を俯瞰したい, so that 「この分野のページが少ない/多い」という感覚を得て次の掘り下げ先を判断できる

#### Acceptance Criteria

1. 知識マップページに「タグ共起」ビューが存在すること
2. ノードはタグであり、ノードサイズはそのタグに属するページ数に比例すること（知識量の濃淡を表現）
3. 同じページに共起するタグ間にエッジを張ること（共起ペアごとに1本）
4. エッジの太さは共起回数（同じタグペアを含むページ数）を反映すること
5. `domains.json` で定義されたドメインタグは視覚的に強調（白枠）すること
6. ノードをクリックすると、そのタグに属するページ一覧が表示されること
7. ページ数2未満のタグは描画対象から除外して視認性を確保すること
8. ズーム・パン操作でインタラクティブに操作可能であること
9. Scrapboxページ本文を表示しないこと（プライバシー保護）
10. ページ詳細APIを呼ばず、既存のKnowledge Proxyのレスポンスのみで描画できること

### Requirement 9: Embeddingsによる知識分布マッピング（Phase 3 将来構想）

**Phase**: 3（将来構想） — データモデル + フロントエンド

**Objective:** As a KJR, I want 知識の意味的な近さを2Dマップで見たい, so that タグやリンクでは見えない知識の「島」や「空白地帯」を発見できる

#### 概要

Scrapboxページのタイトル+descriptionsをembedding vectorに変換し、t-SNEまたはUMAPで2次元に次元削減して散布図として表示する。ハッシュタグベースの分類（Req 2）がトップダウンの分類なのに対し、embeddingsは**ボトムアップに意味的な近さ**を可視化する。

#### Acceptance Criteria

1. Scrapboxページのテキスト（タイトル + descriptions）からembedding vectorを生成できること
2. embedding vectorを2次元に次元削減（t-SNE or UMAP）して座標を得ること
3. 知識マップページに2D散布図ビューが存在し、各点がScrapboxページを表すこと
4. 点はハッシュタグに応じて色分けされること（Req 2のドメイン分類と連動）
5. 点をクリックするとページのメタデータが表示されること
6. 意味的に近いページのクラスタが視覚的に判別可能であること

#### 拡張性に関する設計考慮事項

- Phase 1-2のデータモデル（`KnowledgePageData`型）にembedding座標フィールドを将来追加可能な構造にする
- 知識マップページのレイアウトは、一覧ビュー / グラフビュー / **マップビュー** の3つのビュー切り替えを想定して設計する
- embedding生成はコストのかかる処理のため、ビルド時 or バッチ処理を想定。ランタイムAPIでの毎回生成は避ける

## 非機能要件

### パフォーマンス
- Knowledge Proxyのレスポンスは10秒以内
- フロントエンド表示はProxy応答後2秒以内

### 互換性
- 既存のScrapboxCardList/Carouselに影響を与えないこと
- 既存のuseScrapboxDataフックとの共存が可能であること

### スコープ外

- 参照フレームワークのYAML定義（ギャップ分析はClaude Codeスキルで実施）
- 新しいタグ記法の導入（`[体系/...]` 等）
- カバレッジ算出（比較基準となるフレームワークは管理しない）
- ブログ上でのギャップ分析ビュー
- Scrapboxページの本文表示
- AIによる自動分類
- リアルタイム同期
